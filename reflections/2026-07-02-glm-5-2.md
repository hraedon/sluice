# Session Reflection — 2026-07-02

**Work summary:** Implemented Plan 006 (multi-provider abstraction) — extended sluice from a single umans front to any Anthropic-/OpenAI-compatible upstream via `--provider {umans|anthropic|openai|generic}`. Added a `TruthSource` protocol with three implementations (polled, header-driven, null), an AIMD `AdaptiveRateController` in the pure core, ratelimit header parsing, and `record_response_headers` wiring in the proxy. Two rounds of adversarial review (Kimi K2.7 + GLM 5.2) found 3 critical bugs and 5 medium issues, all fixed. 293 tests pass, CI green on 3.12 + 3.13. Also resolved WI-021 (dashboard breaker_half_open_age_seconds rendering test).

## On the project

sluice is at a mature stage where the pure core / shell separation is well-established and the conventions are strong. Plan 006 was the largest abstraction yet — it introduced a second controller strategy (AIMD) alongside the existing concurrency reconciler, and a provider registry that selects between them. The architecture held up well: the pure core stayed pure (stdlib-only, no clock, no I/O), the import boundary is enforced by test, and the umans regression gate (effective_permits traces identical) passed.

The most interesting design tension is between "the core is pure" and "the AIMD controller needs mutable state" (the current permit level). The solution — threading `AdaptiveSnapshot` through as an argument like `BreakerSnapshot` — is clean and consistent with the existing pattern. The shell holds the snapshot, the core computes the next one.

The dashboard is still inline HTML/JS in `proxy.py` (~260 lines) and should be extracted when it next needs significant changes. The `providers.py` module is new and substantial (~370 lines) but well-structured.

## On the work done

**What went well:** The `UsageReading` → `LimitState` rename with alias preservation was the right call — zero churn in existing code. The provider registry is simple and extensible. The AIMD controller is compact (~60 lines) and its test coverage is thorough (trajectory, monotonicity, hard stops, all decrease triggers).

**What the adversarial review caught (and I'm grateful it did):**
- **C1 (readiness deadlock):** HeaderTruthSource returned `ok=False` when no headers were seen, creating a deadlock where the proxy would never let traffic through to collect headers. Fixed by returning `ok=True` with a conservative `requests_remaining=0` initial state.
- **C2 (NullTruthSource staleness):** The cached reading's `fetched_at_monotonic` was frozen, causing unbounded age growth and permanent AIMD backoff. Fixed by returning a fresh `CachedReading` on each fetch.
- **C3 (429 not pruned):** The `_recent_429s` deque wasn't pruned during `tick()` for the adaptive path, so a single 429 kept the controller in permanent multiplicative-decrease. Fixed by adding `_prune_429s()` at the start of the adaptive path.
- **M1 (low_remaining_fraction dead):** The config field existed but was never used — the controller only checked `<= 0`. Fixed by parsing limit headers and using the fraction check.
- **M3 (empty headers overwrite):** A 500 with no ratelimit headers would overwrite the last good reading with all-None fields, causing the AIMD controller to increase during an error. Fixed by only updating the cache when ratelimit headers are actually present.

**What I'd want a second pair of eyes on:** The `HeaderTruthSource` initial-state design (ok=True, remaining=0) is a deliberate trade-off — it opens the gate so traffic can flow, but the AIMD controller immediately backs off to min_floor. This is correct but means the first few requests against a fresh Anthropic/OpenAI provider will be throttled to 1 permit. That's fail-safe, but operators should know about it.

The `_RATELIMIT_HEADER_ALLOWLIST` constant was removed (it was dead code — the parser picks specific keys directly). If we later want to assert that only allowlisted headers are read, we'd need to reintroduce it as a filter, not just documentation.

## On what remains

- **WI-006 (usage-tap seam)** is explicitly optional and last in the plan's sequencing. It's only needed if observability requires per-request token counts that the provider's own billing API can't supply.
- **Live validation (WI-007 part 2):** Each provider should be validated against its real endpoint per AGENTS.md rule 6. For Anthropic, observe `*-ratelimit-*` headers under normal load and one controlled low-`max_tokens` burst to see a single 429 → back-off, without a 429-storm.
- **Retry-After routing (partially implemented):** The plan's WI-004 says "routes Retry-After into Plan 003's gate_closed_reason / retry_after_seconds." The proxy records response headers (including the `retry-after` header in the raw dict), but `parse_ratelimit_headers` doesn't extract it into `LimitState`, and `gate_closed_reason()` / `retry_after_seconds()` don't consult it. This is a known gap — the header is available in the truth source but not surfaced to the reconcile loop's fast-fail logic.
- **Dashboard extraction** — the inline HTML/JS in `proxy.py` is ~260 lines and growing. Should be extracted to a template file when it next needs significant changes.
- **Prometheus NaN for None values** — `breaker_half_open_age_seconds` emits `# HELP`/`# TYPE` but no sample when the breaker is not half-open. Some consumers expect a sample line.

## Gaps to flag

- **`Provider.auth_extra` is defined but never wired** (`src/sluice/providers.py:309`). The `anthropic-version` header is in the bundle but not sent on any request. For header-driven providers this is harmless (no polling), but it's misleading.
- **`default_429_classifier` is defined but unused** (`src/sluice/providers.py:273`). The proxy hardcodes `_is_concurrency_429()`. If a provider needs a different classifier, this would need wiring.
- **HeaderTruthSource `ok` threshold is hardcoded at 15.0s** (`src/sluice/providers.py:124`), matching the default `AdaptiveConfig.fresh_ttl` but not responding to config changes. If an operator sets `fresh_ttl=60`, the `ok` flag still uses 15s.
- **`ready` for NullTruthSource is True after the first tick** (`src/sluice/reconcile.py:321`). This is correct (no external dependency to wait for), but the startup window is effectively skipped for `generic` providers. The first request may hit an upstream that isn't ready.
- **`test_adaptive_reconcile_loop_backs_off_on_429` uses `FakeHeaderTruthSource`** (`tests/test_multi_provider.py:466`) which manually populates `_cached` rather than going through `record_response_headers`. This tests the AIMD controller's reaction to 429s but doesn't exercise the full header→truth→controller pipeline.
