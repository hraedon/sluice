# Session Reflection — 2026-07-08

**Work summary:** Deep adversarial code review across all source files (4 subagents covering control/reconcile, proxy/admin, gate/session/trust/singleton/metrics, and test coverage), followed by targeted fixes for 9 issues across 6 source files + 3 test files, adversarial review of the fixes (which caught a test-file corruption bug), and release of v1.2.3.

---

## On the project

sluice is in a mature, production-deployed state. The codebase is clean and well-structured — the deterministic-core / thin-shell split is enforced, the AGENTS.md hard rules are specific and load-bearing, and the test suite (781 tests) is comprehensive. The v1.2.2 release added the WI-024 rate_limit 429 fix (rate_limit 429s no longer feed the breaker), and this session caught several remaining fail-open paths and correctness issues that the WI-024 work exposed.

## On the work done

The multi-model adversarial review was effective. Four subagents (GLM, Kimi, Minimax, explore) covered different parts of the codebase and found 9 actionable issues out of ~60 total findings. The key wins:

1. **Stale-reading hard_cap clamp (B1)** — the most significant find. When the `/v1/usage` reading goes stale, the code previously dropped the `hard_cap` clamp entirely, which was fail-open on provider downgrade during a poll outage. The fix is one line: use `min(config.target, reading.hard_cap)` as the clamp on both fresh and stale paths. The existing test encoded the wrong behavior; it was renamed and the assertion flipped.

2. **Adaptive controller safety net (F1)** — the stale-reading + rate_limit 429 safety net was only in the concurrency-reconcile branch. The adaptive controller (Anthropic/OpenAI/generic) had no equivalent. This was a real fail-open window for non-umans providers.

3. **Idle detection (F3)** — `_idle` didn't check `_recent_rate_limit_429s`, causing poll-cadence oscillation when rate_limit 429s were arriving.

4. **RFC 7230 Connection header parsing** — the proxy didn't parse the `Connection` header for additional hop-by-hop headers, which is a cache-transparency gap (AGENTS.md rule 7).

5. **Session fixation prevention** — upstream `Set-Cookie: sluice_session=...` was not stripped from responses.

6. **Prometheus label escaping** — `metrics.py`'s `to_prometheus()` didn't escape labels (status.py did). Minor but inconsistent.

7. **Gate negative resize guard** — `resize()` accepted negative values. Defensive fix.

8. **IPv4-mapped IPv6 loopback** — `_should_set_secure()` missed `::ffff:127.0.0.1`.

9. **`min_floor` validation** — `ControllerConfig` and `AdaptiveConfig` didn't validate `min_floor >= 1`. A `min_floor=0` would void the "never fully closed" guarantee.

The adversarial review of the fixes (Nemotron 3 Ultra) caught a test-file corruption bug: my edit accidentally dropped the function definition for `test_rate_limit_429_does_not_retrip_half_open_breaker()`, leaving its body as dead code inside the preceding test. This was caught and fixed before the release.

## Gaps to flag

- **Connection header parsing doesn't handle quoted tokens** (RFC 7230 edge case). Standards-compliant peers won't trigger this.
- **`_should_set_secure` IPv6 loopback check is string-based** — could use `ipaddress` module instead for full correctness. Only `::ffff:127.0.0.1` was added; other IPv4-mapped forms (`::ffff:0:127.0.0.1`, `::ffff:7f00:1`) are still missing.
- **Adversarial review found several low-priority findings not addressed** — login throttle global DoS (S1), K8s lease 409-on-create race (K1), audit logging uses peer IP not forwarded IP (L-1). These are documented and can become work items if prioritized.
