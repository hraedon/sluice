# Session Reflection — 2026-07-04

**Work summary:** Implemented Plan 012 (dashboard login page) end-to-end — all seven work items: pure session primitives in a new `session.py` module, `/login` + `/logout` routes, cookie-based auth acceptance with `WWW-Authenticate` removal, CSRF fetch-metadata check + Rule-7 cookie stripping, global login throttle, home-screen manifest/icon polish, and docs. Also resolved WI-027 (httpx `timeout=None` → finite connect/read/write/pool) and closed WI-021 (already resolved in git). ## On the project

sluice's architecture made this change clean to land. The pure-core / thin-shell split meant session primitives could live in `session.py` as pure functions with `now` passed as an argument — no clock, no I/O, fully unit-testable without ASGI. The import-boundary test (now extended to cover `session.py`) enforces this at CI level. The shell modules (`admin.py`, `proxy.py`) did what shells do: route, delegate, strip headers, read the clock. No business logic crept into the proxy layer.

The auth surface is now three-pronged (Bearer, Basic, session cookie) but the acceptance path is a single function (`check_admin_auth`) — every admin route calls it, and the CSRF check is layered on top for mutations only. This is the right shape: the security decision is centralized, the route handlers are thin. The plan was unusually detailed (191 lines for a login page), which meant implementation was mostly transcription. The `§4` Secure-cookie logic — auto-set on https/localhost, omitted with a warning on plain-HTTP LAN — is the kind of detail that would have been a prod incident if left to "just set Secure always."

The dashboard's inline JS is now ~800 lines. It's the right call for a home-lab tool (no build step), but the logout button and 401-redirect were the first dashboard changes that touch the *control flow* of the page rather than just rendering. A JS test harness is increasingly overdue.

## On the work done

**What went well:** Delegating WI-001 + WI-005 (pure functions) to a GLM subagent was the right call — it came back with clean, well-tested code in one shot, and I could verify it independently while working on the security-critical route changes in parallel. The adversarial review earned its cost: both reviewers independently found the cookie-stripping inconsistency (the proxy split on `"; "` but the auth parser split on `";"`), which would have leaked `sluice_session` upstream on non-standard Cookie headers — a direct Rule-7 violation. The `read=None` timeout was also flagged by both — I had set `connect=10.0` but left `read=None`, which means a hung upstream mid-stream would still hold a permit forever. Now `read=300.0` (5 minutes between chunks — generous for LLM streaming, tight enough to catch a hung connection).

**What I'm confident in:** The session cookie construction (HMAC-SHA256 keyed off `sha256(b"sluice-session-v1" + admin_token)`) is sound — no new secret, token rotation revokes everything, constant-time verify. The CSRF check (SameSite=Strict + Sec-Fetch-Site for cookie-auth mutations, Bearer exempt) is correct per the plan. The `/logout` now returns 405 on non-POST methods (was falling through to the proxy). The login throttle is global (per-IP is theater behind an ingress). The `X-Forwarded-Proto` multi-value case is handled (first value wins).

**What I'd want a second pair of eyes on:** The 0.2s `asyncio.sleep` on login failure is a blunt instrument against timing side-channel. It's not a true constant-time delay (success still returns immediately), and 0.2s is arbitrary. A real solution would be a pre-computed delay or a minimum response time guarantee. Also, the cookie-stripping logic in `_filter_request_headers` now splits on `";"` and strips each part — this is correct but re-serializes the remaining cookies with `"; "` separator, which technically changes byte layout if the original used a different separator. In practice browsers always use `"; "`, so this is fine, but it's not byte-for-byte transparent for non-browser clients with unusual Cookie formatting.

**Deliberate trade-offs:** I did not add CSRF protection to `/logout` (the plan scopes CSRF to "mutation handlers"). `SameSite=Strict` blocks cross-site cookie transmission, so a cross-site logout form can't carry the session cookie — but the server still sends `Set-Cookie: Max-Age=0`, which the browser applies and could delete an existing cookie. This is annoyance-level DoS, not a data breach, and matches the plan's scope. I also did not add `X-Frame-Options` / CSP `frame-ancestors` headers — the login page reveals nothing (the dashboard behind it is counts-only), but clickjacking protection is a reasonable hardening step.

## Gaps to flag

- **Login failure timing side-channel** (`admin.py:316`): 0.2s `asyncio.sleep` is not true constant-time — success returns immediately. An attacker measuring response time can distinguish valid from invalid tokens under the throttle threshold. The token is 64 hex chars, so brute-force is impractical regardless, but the side-channel exists.
- **No `X-Frame-Options` / CSP `frame-ancestors`** on login or dashboard pages (`admin.py:send_login_page`, `admin.py:send_dashboard`): clickjacking is possible. The login page reveals nothing, but UI-redressing against the dashboard is conceivable. Low risk for a single-operator tool.
- **Cookie re-serialization** (`proxy.py:625`): filtered Cookie headers are re-joined with `"; "`, which is standard but not byte-transparent if the original used a different separator. Browsers always use `"; "`, so this is a non-issue in practice.
- **`doLogout` form submission** (`dashboard.html:225`): creates a hidden form and submits it to `/logout`. This works but is unusual — a `fetch(POST)` with `credentials:'include'` would be more conventional. The form approach was chosen because it triggers a full page navigation (back to `/login`), which is the desired UX.
- **No test for `read=300.0` timeout**: the timeout is set but no test verifies that a hung upstream actually triggers it and releases the permit. An integration test with a mock transport that never sends data would close this gap.
- **Session cookie on plain-HTTP LAN**: the warning is logged but the cookie is still set without `Secure`. This is by design (§4 of the plan — degrading visibly beats breaking silently), but it means the session cookie is sniffable on the LAN. The README documents this; the code logs a warning. Acceptable for the documented Docker quickstart path.
