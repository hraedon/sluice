## On the project

sluice's architecture is clean and well-documented. The pure-core / async-shell split enforced by the import-boundary test is the right shape — control.py is genuinely pure (no clock, no I/O), and the proxy/gate/reconcile shell wraps it cleanly. The breaker state machine is more subtle than it looks — the interaction between event-driven transitions (record_429/record_success) and tick-driven transitions (breaker_on_tick) creates a real concurrency window during the fetch's await, and the WI-016 fix (reading self._breaker after the fetch) is the correct approach. The concurrent breaker test now exercises this with real async.

## On the work done

**What went well:** The p95 formula tests are clean — directly populating `_wait_samples` is the right approach for testing the formula (vs. the `_grant_after_wait` helper which has a 10s fake-clock timeout that breaks for elapsed > 10). The adversarial review was genuinely useful — it caught a major flakiness risk in the normal-request test and several minor issues (BaseException breadth, get_event_loop vs get_running_loop, unreachable return, asymmetric cleanup).

**What was hard:** The integration disconnect test was the hardest part. The original test had a weak assertion (`cancelled OR received`) that was always true. Fixing it to require actual upstream termination required understanding a subtle HTTP/1.1 + uvicorn limitation: after the request body is complete, uvicorn doesn't actively monitor the socket for client disconnects. The `http.disconnect` event is only delivered when the ASGI app calls `receive()`. So the upstream handler must alternate `send()` and `receive()` (with a short timeout) to give uvicorn a chance to deliver the disconnect event. This is the same pattern the proxy itself uses (disconnect_watcher task). This took several iterations to get right.

**What I'd want a second pair of eyes on:** The integration test's 0.05s timeout on `receive()` in the upstream handler — if uvicorn is slow to deliver the disconnect event, the test could still pass via the `except (Exception, asyncio.CancelledError)` path if the send() eventually fails. But the test validates the right thing: the upstream was terminated, not completed.

## Gaps to flag

- **HTTP/1.1 disconnect detection limitation:** The proxy closes the upstream TCP connection on client disconnect (`stream_cm.__aexit__()`), but the upstream may not detect this until it tries to write and the TCP buffer fills. The integration test works around this by having the upstream call `receive()` periodically. In production, the upstream (umans API) handles this at its own level — sluice's job is to close the connection, not to guarantee the upstream notices.
- **Concurrent breaker test timing:** The test now waits for `client.fetch_count > 0` before recording the 429, which is better than a fixed sleep. But on extremely loaded runners, the fetch might complete before the 429 is recorded, making the test exercise a non-concurrent path. The test still passes (breaker trips either way) but may not always test what it claims.
- **Integration test could be slow on CI:** The upstream sends 100 chunks with 0.05s receive-timeout per chunk = ~5s worst case. On a loaded runner, this could approach the 15s timeout.
