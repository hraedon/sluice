## On the project

sluice is maturing into a real in-path proxy. The Plan 012 work added a proper session cookie layer (session.py) with HMAC-signed cookies, CSRF guards, and a login throttle — the security surface is now appropriate for a tool that sits in the live request path. The separation of concerns (session.py is stdlib-only, admin.py handles routes, proxy.py handles forwarding) is clean and the import-boundary test enforces it.

The main tension I see is the `read=None` timeout decision. The prior session (glm-5-2-2) explicitly set `read=300.0` after adversarial review flagged `read=None` as a permit-leak risk. This session's user explicitly asked for `read=None` back, and both adversarial reviewers flagged it again. I honored the user's request with a detailed comment explaining the tradeoff, but this is a genuine design tension that hasn't been fully resolved. The right long-term fix is an application-level liveness watchdog around the streaming loop, not a read timeout.

The dashboard JS is getting complex enough that source-level string assertions are becoming fragile. Two reviewers noted this. A JS test harness is increasingly overdue.

## Gaps to flag

- **`read=None` permit-leak risk** (`proxy.py:99`): A hung upstream (silent TCP death) holds a gate permit indefinitely. The only recovery is client disconnect or drain timeout. Both adversarial reviewers flagged this as a regression from the prior `read=300.0`. The comment documents the tradeoff but doesn't resolve it. If hung upstreams become a problem, an application-level liveness watchdog is needed.
- **`_should_set_secure` still trusts XFP** (`admin.py:156`): The docstring now warns about the risk, but the code still trusts any `X-Forwarded-Proto: https` header. A misconfigured intermediate hop on a plain-HTTP origin causes a silent login loop. Uvicorn's `--forwarded-allow-ips` should be configured to restrict this, but that's a deployment concern, not a code fix.
- **Login throttle is global** (`session.py:73`): With `max_failures=20` and `lockout_seconds=120`, a LAN peer can still deny login to everyone for 2 minutes with 20 bad attempts. The defaults are more lenient now but the fundamental tradeoff (global, not per-IP) remains because ingress masks IPs.
- **Dashboard JS tests are source-level string assertions** (`test_plan012.py:TestDashboard403Behavior`): Checking for specific strings in HTML is fragile to refactoring. A JS test harness would be more robust.
- **WI-021 stuck in review**: The test exists and passes, but the lifecycle prevents self-review closure. Needs a human or different-model reviewer.
