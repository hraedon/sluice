## On the project

This was my second session in sluice today, so the codebase was familiar. The
architecture is clean: a pure stdlib-only core (`control.py`) that makes all
the concurrency decisions, wrapped by an async shell (proxy, reconcile, usage,
singleton, gate, status, cli). The import-boundary test enforces the purity
contract mechanically. The concurrency model doc is the design spine — it's
genuinely useful, not just documentation theater. I read it first and it
oriented me immediately.

The singleton guard design is the most interesting piece. It's a distributed
systems problem (leader election via k8s Lease) reduced to a small interface
with a no-op default for local dev. The split-brain risk (WI-003) was the
right thing to flag as high priority — without local lease expiry checking, a
pod whose event loop stalls for >30s would happily serve traffic as a phantom
leader, and the only detection was the renewer eventually failing. That's a
correctness bug, not a performance one.

One thing that struck me: the reconcile loop's `tick()` method is doing a lot
— fetch, breaker transition, phantom windowing, effective_permits, gate resize,
state caching. It's well-structured but it's the kind of method where a subtle
ordering bug (e.g., reading `self._breaker` before vs after the `await
self._usage.fetch()`) could cause subtle race conditions. The existing comment
about reading `self._breaker` after the fetch is a good sign that someone was
thinking about this.

## On the work done

The five fixes themselves were straightforward — each was a small, focused
change:

- **WI-003** (is_held local expiry): Added `_last_renewed_mono` tracked on
  acquire/renew, checked in `is_held()`. The trickiest part was deciding
  whether to clear `_held` in `is_held()` — I chose not to, so the renewer can
  still recover (a successful renew flips `is_held()` back to True without
  needing a full re-acquire).

- **WI-004** (retry acquire): A background task that periodically retries
  `guard.acquire()` when the initial startup acquire fails. The adversarial
  review caught the real bug here (see below).

- **WI-005** (renew_interval validation): One-line `ValueError` in `__init__`.
  Simple and correct.

- **WI-006** (stale resets_at cap): Cap at 300s when `cached.ok=False`. The
  question of whether this "loosens" the gate is interesting — it reduces
  Retry-After, which means clients come back sooner, but the gate itself stays
  closed. I think this is the right call: a stale 5-hour Retry-After is
  misleading, and sluice still rejects with 503.

- **WI-007** (bounded body queue): `asyncio.Queue(maxsize=1)`. The simplest
  fix with the most leverage — one keyword argument turns unbounded memory
  growth into natural backpressure.

**What the adversarial review caught:** The Kimi reviewer (GLM returned empty)
found that `_guard_acquired` was set to `True` *before* `start_renewer()` and
`reconcile.start()` were called. If either raised, the `while not
_guard_acquired` loop would exit (condition already false), leaving the lease
held with no renewer or reconcile — a zombie leader serving traffic without
usage polling or lease renewal. This was a real bug that my tests didn't catch
because I only tested the happy path. The fix moves the flag assignment after
both calls succeed, with a try/except that releases the lease and continues
retrying on failure. I added a test (`test_retry_acquire_releases_lease_if_start_fails`)
that specifically exercises this path.

**What I'm confident in:** WI-003, WI-005, WI-006, WI-007 are all small and
well-tested. The is_held() local expiry check is the most consequential change
and I'm satisfied with the design — the boundary is exclusive (`>`), consistent
with the server-side `_is_expired` which also uses `>`.

**What I'd want a second pair of eyes on:** The WI-004 retry-acquire path is
the most complex change, and the lifespan integration is only tested
indirectly (I test `_retry_acquire` directly, not through the full ASGI lifespan
flow). A full lifespan integration test would be valuable but is complex to
write without a real ASGI server. Also, the `_retry_acquire` method doesn't
have backoff/jitter — a fixed interval is fine for a home-lab scale, but under
adversarial conditions (many pods retrying simultaneously) you'd want jitter.

## On what remains

   AGENTS.md rule #6. The bounded body queue (WI-007) in particular could

2. **Consider exposing `retry_interval` via CLI/env.** Currently it's hardcoded

   production tuning this should be a flag.

3. **Consider backoff/jitter in `_retry_acquire`.** Fixed interval is fine for
   now but could cause thundering herd if multiple pods are retrying.

4. **Full lifespan integration test for WI-004.** The current test exercises
   `_retry_acquire` directly but doesn't verify the full startup→retry→shutdown
   flow through the ASGI lifespan handler.

Nice-to-have:
- The `is_held()` local expiry check could log when it triggers (it currently
  silently returns False). A log line would help operators understand why a pod
  stopped serving.
- The `_RETRY_AFTER_STALE_CAP` constant (300s) is somewhat arbitrary. It might
  be worth making it configurable or at least documenting the rationale.

## Gaps to flag

- **`_retry_acquire` has no backoff or jitter** (`proxy.py:188`). Fixed 10s
  interval could cause thundering herd if multiple non-leader pods retry
  simultaneously after a leader crash. Low priority at home-lab scale.

- **`retry_interval` not exposed via CLI** (`cli.py`). The `ProxyApp.__init__`
  accepts `retry_interval` but `cli.py:_cmd_serve` doesn't pass it, so it always
  uses the default. Minor, but means the parameter is untestable in production
  without code changes.

- **Tests for WI-004 test `_retry_acquire` directly, not through lifespan**
  (`test_proxy.py:842`). The lifespan handler's retry-task creation and
  shutdown cancellation paths are not exercised by any test. The direct tests
  verify the helper logic but not the integration.

- **`is_held()` silently returns False on local expiry** (`singleton.py:271`).
  No log line is emitted when the local lease expiry check triggers. An operator
  seeing `/readyz` flip to 503 would have to infer the cause from the renewer's
  subsequent failure logs, not from the expiry check itself.
