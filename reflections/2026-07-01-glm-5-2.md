# Session Reflection — 2026-07-01

**Work summary:** Reviewed the sluice project and the external umans-dash repo. Fixed two identified gaps: WI-008 (dashboard/admin auth mutual exclusivity — added Basic auth support, gated the dashboard route, stripped admin credentials from proxied requests) and the `_BODY_QUEUE_MAXSIZE = 1` disconnect-detection delay (eliminated the queue entirely, replaced with direct `receive()` + `disconnect_watcher`). Added three dashboard UI features: visual concurrency gauge, sparkline, and config readout. Ran three independent adversarial reviewers across the whole project, filed 8 new work items (WI-013 through WI-020) for pre-existing issues found.

---

## On the project

sluice is the most architecturally disciplined project I've worked in. The pure-core/async-shell split isn't just convention — it's enforced by a test. The concurrency model doc is genuinely the design spine; you read it first and everything follows. The "inert in-path" guarantee (never read, log, store, or rewrite bodies) is unusual for a proxy and creates a sharp constraint that eliminates whole categories of features (no caching, no routing, no body transformation) but makes the cache-transparency rule (Rule 7) achievable.

The weakness is in async lifecycle edge cases — the areas that are hardest to test locally and most likely to break in production. The adversarial review surfaced three critical pre-existing bugs (double `http.response.start`, upstream not cancelled on body-upload disconnect, half-open clamp raising permits) that all stem from the difficulty of reasoning about ASGI/httpx interaction under disconnect and error conditions. These are the classic "works on the happy path, breaks under load" issues.

## On the work done

**What went well:** The Basic auth + header stripping fix was clean — the reviewer's catch about browser Basic auth leaking origin-wide was the kind of thing that would have been a real security incident in production. The queueless `body_stream()` refactor is a genuine improvement: simpler code, no memory buffering, no blocking. The dashboard UI features are pure client-side JS with no Python changes (except the config field), which is the right boundary.

**What I'm less confident in:** The `disconnect_watcher()` design has a known gap — if the client disconnects during body upload, `body_stream()` returns but httpx still sends the truncated request upstream. I flagged this as WI-014 but didn't fix it because the fix requires wrapping the httpx stream in a cancellable task, which is a significant refactor. The old pump+queue had the same behavior, so it's not a regression, but it's still a Rule 4 violation.

**What I'd want a second pair of eyes on:** The `esc()` HTML-escaping helper in the dashboard JS. It's correct in principle (creates a span, sets textContent, reads innerHTML), but I'd want someone to verify it's used in every place where dynamic data reaches innerHTML. The SVG innerHTML assignments use only numeric data, which should be safe, but SVG has different parsing rules than HTML.

## On what remains

The 8 filed work items (WI-013 through WI-020) represent the pre-existing issues found by the adversarial review. In priority order:

1. **WI-015 (critical):** Half-open breaker clamp raising permits from 0 to 1 — this is a pure-core bug in `control.py` that directly violates the fail-safe guarantee. It's a one-line fix (change the clamp floor) but needs a test with `phantom_estimate > 0 AND breaker=HALF_OPEN`.
2. **WI-013 (critical):** Double `http.response.start` — needs a `response_started` flag. Moderate complexity.
3. **WI-014 (critical):** Upstream not cancelled on disconnect — the most architecturally significant fix. Requires wrapping the httpx stream in a cancellable task.
4. **WI-016 (high):** Breaker race — needs a locking strategy. Three options were outlined in the WI body.
5. **WI-017 (high):** Phantom sample timing — simple fix (capture `gate.held` before `await fetch()`).
6. **WI-018 through WI-020 (medium):** Startup fail-open window, 429 type discrimination, probe timeout.

Additionally, four simpler issues were identified but not filed (good subagent candidates): K8s lease leak on cancellation (add `except CancelledError`), `UsageClient.close()` not called on shutdown, shutdown not resilient to errors, and admin routes public by default.

## Gaps to flag

- **No test for disconnect during body upload** — `test_permit_released_on_disconnect` only tests disconnect during response streaming. The body-upload disconnect path is untested (`tests/test_proxy.py`).
- **No test for `aiter_raw()` raising after headers sent** — the double `http.response.start` bug (WI-013) has zero test coverage.
- **No test for concurrent `record_429` during `tick()`** — the breaker race (WI-016) is explicitly untested.
- **`_AsyncMockTransport` in tests calls `await request.aread()` which buffers the entire body** — this means no test actually exercises the streaming/backpressure behavior of `body_stream()`. The backpressure claim in the docstring is untested.
- **The dashboard `fetch('/status.json')` with `credentials:'same-origin'`** — should work in browsers but has no browser-level test. The `test_dashboard` test only checks that the HTML is served, not that the JS executes correctly.
- **`p95_wait_seconds` calculation in `gate.py:138`** uses a ceil-based index that underestimates p95 for small samples (n=2 gives the 50th percentile). Not a bug per se, but worth documenting.
