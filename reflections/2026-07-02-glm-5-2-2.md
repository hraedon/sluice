# Session Reflection — 2026-07-02

**Work summary:** Comprehensive project review followed by fixing all identified bugs and gaps in a single sweep. 5 parallel agents fixed code bugs, 2 agents filled test gaps, 3 adversarial reviewers found 6 additional issues which were then fixed. 18 files changed, +1055/-110 lines, 383 tests passing, CI green on 3.12+3.13. PR #3 open.

## On the project

Sluice is architecturally clean — the pure-core/shell separation is well-enforced, the concurrency model is principled, and the test suite is genuinely thorough on the hard parts (disconnect races, streaming, phantom absorption). The deployment is solid GitOps with good pod hardening.

The main structural weakness is that the multi-provider abstraction (Plan 006) was added without fully updating all the seams it created. `soak.py` was broken by the `usage_client`→`truth_source` rename. `auth_extra` and `default_429_classifier` were left as dead code. The `HeaderTruthSource` freshness threshold was hardcoded instead of config-driven. These are all "the last 10% wasn't done" issues — the plan was sound, the execution just didn't fully propagate.

The dashboard inline HTML/JS (~270 lines in proxy.py) is a growing maintainability concern. It's been noted in prior reflections but never extracted. Each dashboard feature addition makes the extraction harder.

## On the work done

The parallel-agent approach worked well for the initial fix phase — 5 agents touching different files had zero conflicts. The adversarial review phase was genuinely valuable: it found a critical bug (AIMD `last_decrease_monotonic` silently reset to 0.0 on healthy ticks, making the rate-limiting non-functional), two critical deployment issues (ServiceMonitor couldn't find the Service due to missing labels, NetworkPolicy blocked Prometheus), and a medium correctness issue (wall-clock vs monotonic for adaptive rate-limiting). These would have shipped without the review.

What I'm confident in: the proxy fixes (429 recording, stream cleanup), the singleton renew recovery, the breaker cooldown fix, the `retry_after` ceil fix, the CLI hardening, the dead code removal.

What I'd want a second pair of eyes on: the AIMD rate-limiting interaction. The `min_decrease_interval=30s` constant is a guess — it should probably be validated against real AIMD oscillation patterns. The `nan` for Prometheus None values is technically correct but some older Prometheus versions may not handle it well. The `SluicePermitLeak` alert expression (`delta(sluice_in_flight[5m]) == 0`) could still false-positive if exactly one long-running stream holds a permit for >5m with no other traffic.

## Gaps to flag

- `scripts/soak.py` still has 2 pre-existing mypy errors unrelated to this session's fix (the agent noted them but didn't fix them since they were pre-existing).
- `control.py` has two "last decrease" fields (`last_decrease_at` and `last_decrease_monotonic`) with subtly different semantics — `last_decrease_at` is written but never read except in tests. Dead state that could confuse future contributors.
- `HeaderTruthSource` initial reading returns `requests_remaining=0` with `ok=True`, causing AIMD to immediately back off to `min_floor=1` on startup against Anthropic/OpenAI. This is documented but operators should be aware of the cold-start throttle.
- `NullTruthSource` (generic provider) sets `ready=True` after the first tick with no external check — the startup window is effectively skipped, meaning the first request may hit an unprobed upstream.
- The `PrometheusRule` `release: kube-prometheus-stack` label assumes a specific Helm deployment name. If the target cluster uses a different release name, neither the ServiceMonitor nor the alerts will be picked up.
- The `SluiceUnreachable` alert uses `job="sluice"` but the actual job label depends on Prometheus configuration and may be formatted differently (e.g., `sluice/sluice`).
- `breaker_on_success` preserves stale `opened_at` when transitioning HALF_OPEN→CLOSED. Currently harmless (only consulted in OPEN state) but could confuse debugging.
