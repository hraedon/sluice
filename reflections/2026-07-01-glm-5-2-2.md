# Session Reflection — 2026-07-01

**Work summary:** Fixed 8 bugs (WI-013 through WI-020) across the pure core (`control.py`), the reconciliation loop (`reconcile.py`), and the async reverse proxy (`proxy.py`). Three critical fixes: double ASGI `http.response.start` on mid-stream upstream drop, upstream request not cancelled on client disconnect, and half-open breaker clamp raising permits from 0 to 1. Two high: breaker race in `tick()` overwriting event-driven transitions, and phantom sample timing mismatch. Three medium: fail-open at startup, recording all 429s as concurrency 429s, and missing HALF_OPEN→OPEN probe timeout transition. Also closed WI-009 through WI-012 (duplicates of WI-013 through WI-016). 17 new tests, adversarial review passed with all findings addressed.

## On the project

sluice is a well-architected concurrency governor with a clean pure-core/shell separation. The concurrency model doc (`docs/concurrency-model.md`) is genuinely excellent — it makes the decision logic transparent and the fail-safe invariants checkable. The codebase is small enough to hold in your head, which is the right state for a safety-critical in-path tool.

The pure core (`control.py`) is the strongest part: deterministic, testable, no I/O. The breaker state machine is simple but correct — three states, clear transitions. Adding `half_opened_at` for the probe timeout (WI-020) was a natural extension that didn't complicate the model.

The proxy (`proxy.py`) is the most complex and fragile part. The `_forward` method was already intricate before this session (body streaming, disconnect watching, phantom prevention), and the WI-014 fix (racing stream entry against disconnect) added another layer of complexity. The manual `__aenter__`/`__aexit__` management is correct but harder to reason about than the original `async with`. This is inherent to the problem — you can't cancel an `async with __aenter__` without manually managing the context — but it's the part I'd want the most eyes on.

## On the work done

**What went well:**
- The control.py fixes (WI-015, WI-016, WI-020) were clean and the tests write themselves — pure functions, deterministic, no async.
- The adversarial review caught real issues: the `http.response.start` send not being wrapped in try/except (a narrow but real double-start edge case), `except BaseException` swallowing `KeyboardInterrupt`, and a missing test for the disconnect-while-waiting-for-headers scenario.
- All 172 tests pass, ruff + mypy clean.

**What I'm confident in:**
- WI-015 (half-open clamp): the math is simple — `max(0, min(base, 1))` never raises above base.
- WI-016 (breaker race): resetting `opened_at` when OPEN is the correct fix — it prevents `breaker_on_tick` from seeing stale cooldown data.
- WI-017 (phantom sample timing): capturing `held_at_fetch` before the fetch is obviously correct — the observed value reflects the fetch time, not post-fetch.

**What I'd want a second pair of eyes on:**
- WI-014 (upstream cancel on disconnect): the racing logic with `asyncio.wait` is complex. The `except (asyncio.CancelledError, Exception)` pattern is correct but the interplay between `entry_task`, `disconnect_task`, and `stream_cm.__aexit__` has many edge cases. The test for disconnect-while-waiting-for-headers exists but uses a mock transport — real-world behavior with httpx's connection pooling might differ.
- WI-019 (retry-after heuristic): using the presence of `retry-after` to distinguish rate-limit from concurrency 429s is a reasonable heuristic but not guaranteed. If the umans API sends `retry-after` on concurrency 429s, the breaker silently stops tripping. This is a calculated tradeoff — the reconciliation loop provides a backstop — but it should be revisited when we learn the actual API behavior.

## On what remains

- **Push and watch CI (3.12 + 3.13).** The async/streaming changes are exactly the kind that are locally-green and actually-broken. AGENTS.md rule 6 is explicit about this.
- **Test the WI-014 racing logic against a real httpx connection**, not just the mock transport. The mock responds immediately; a real upstream with latency is where the racing matters.
- **Verify the `retry-after` heuristic** against the actual umans API. If concurrency 429s include `retry-after`, we need a different distinguishing signal.
- **Consider adding `half_opened_at` to the status/dashboard.** It's currently invisible — the dashboard shows breaker state but not how long it's been half-open.

## Gaps to flag

- **WI-019 `retry-after: 0` edge case**: `response.headers.get("retry-after")` returns `"0"` (truthy), so the 429 is NOT recorded. If the provider sends `retry-after: 0` on concurrency 429s, this is a silent fail-open. `proxy.py:423`. Test exists documenting this behavior.
- **`stream_cm.__aexit__` called with `(None, None, None)`** even when an exception occurred. httpx's `__aexit__` just calls `aclose()` regardless, so this is fine today, but if httpx changes this behavior, the proxy could leak connections. `proxy.py:459-465`.
- **`_make_app` test helper sets `_first_poll_ok = True`** to work around the WI-018 fast-fail. This means most proxy tests don't exercise the startup window. Tests that specifically need "not ready" behavior must override it. `tests/test_proxy.py:121`.
- **The `disconnect_watcher` and `body_stream` share `receive()`** sequentially, not concurrently. If the ASGI server sends `http.disconnect` while `body_stream` is blocked on `receive()` for body chunks, the disconnect is detected. But if the body is complete and `disconnect_watcher` hasn't started yet (race between `body_done.set()` and `disconnect_watcher`'s `await body_done.wait()`), a disconnect could be missed. This is a very narrow window but worth noting. `proxy.py:360-366`.
