# Session Reflection — 2026-07-04

**Work summary:** Evaluated and implemented feedback from two external code reviewers (GPT-5.4 and Gemini) across three areas: (1) refactoring proxy.py by extracting lifecycle management to a new module, (2) fixing a verified fail-open bug in the 429 breaker classification, and (3) adding graceful connection draining on shutdown. Two rounds of adversarial review caught a critical SQLite persistence gap, a race condition in the drain path, and a pre-existing CancelledError lease leak. Single commit 332d578, 483 tests pass.

## On the project

sluice is maturing well. The pure-core / thin-shell architecture held up under three simultaneous changes — none of them touched control.py's logic, and the import boundary test caught the new lifecycle module needing to be in SHELL_MODULES. The capture doc (docs/wi-024-429-capture-2026-07-03.md) was the key artifact: it turned a reviewer's hypothesis ("maybe umans sends retry-after on concurrency 429s") into a verified fact with a log line, a status snapshot, and a timeline. Without that doc, the fix would have been speculative. With it, the fix was obvious.

The 429 classification story is messy but honest. The retry-after heuristic was always documented as an unverified assumption — the code even had a comment saying "Revisit when a real concurrency 429 is observed live." The capture doc is that observation. The fix (feed everything non-CDN to the breaker) is deliberately coarse: it trades false-trip risk for fail-safe guarantee. Provider-specific classifiers (Plan 006) would refine this, but the coarse approach is correct for now — the breaker threshold (5 in 5 minutes) prevents single events from tripping.

## On the work done

**What went well:** The adversarial reviews earned their cost. The first round caught the import boundary gap and a pre-existing CancelledError lease leak. The second round caught a critical SQLite persistence gap — the rate_limit_429s field was added to HistoryEntry but not to the SQLite schema, INSERT, SELECT, or load_recent. That would have silently dropped the field on any deployment using --history-store. The third round caught a race condition where a request blocked on gate.acquire() during drain is invisible to the drain loop (it's a waiter, not a holder) and would hang until queue_timeout after the client was already closed.

**What I'm confident in:** The lifecycle extraction is a clean separation — ProxyApp delegates lifespan events and keeps its own guard/reconcile references for request-path checks. The 429 fix is the right call given the capture data. The drain implementation with its try/finally guard and post-acquire race check is solid.

**What I'd want a second pair of eyes on:** The while/else drain loop is correct but subtle — Python's while-else runs the else block only when the loop condition becomes False (not on break). The `drained` flag variable makes this explicit, but a reader unfamiliar with while/else might miss the semantics. Also, the drain polls at 100ms intervals rather than using gate._cond.notify_all() — functionally correct but wasteful. A `gate.wait_until_empty(timeout)` method would be cleaner but would add API surface to the gate for a shutdown-only concern.

**Deliberate trade-offs:** I skipped two of GPT-5.4's six suggestions. The gate.py cooldown precision concern was wrong — the existing code is already O(1) amortized (sorted deque, pop from head). The admin.py inline HTML concern was also wrong — the HTML template is already external at static/dashboard.html. I evaluated and rejected these explicitly rather than silently ignoring them.

## On what remains

- **WI-024** (open, medium): The 429 classification issue is partially addressed. The fail-open is fixed, but the false-trip concern (all non-CDN 429s now feed the breaker, including genuine rate-limit 429s) remains. Provider-specific classifiers would resolve this.
- **WI-027** (open, medium): Upstream httpx client has timeout=None — no connect timeout. The graceful drain helps with shutdown but doesn't address an unresponsive upstream during normal operation. Permits are held indefinitely.
- **WI-028** (open, low): QoS header spoofing — filed this session from Gemini's feedback. Design trade-off for trusted deployments.
- **Provider-specific 429 classifiers** (Plan 006 follow-up): Bind classification heuristics to provider config so umans can have different rules than Anthropic/OpenAI.
- **Drain improvement**: Replace 100ms polling with gate condition notification for instant wake on last permit release. Low priority — the polling works correctly within the timeout budget.

## Gaps to flag

- **reconcile.stop() closes the truth source before drain completes** (reconcile.py:480): `reconcile.stop()` calls `self._truth.close()`. In-flight requests during drain may call `reconcile.record_response_headers()` → `truth.record_response_headers()`. Currently safe (all truth sources have no-op or independent record methods), but a future truth source with non-trivial close() could break. Consider splitting stop() into stop_loop() and close_truth().
- **Drain loop doesn't use gate condition** (lifecycle.py:140): Polls at 100ms instead of waiting on `gate._cond`. The gate's `release()` calls `notify_all()` but the drain loop doesn't wait on it. Functionally correct, wasteful.
- **WI-019 reversal is deliberate but undocumented in code**: The fix deliberately reversed WI-019's behavior (rate-limit 429s now feed the breaker again). The WI-024 append-body note documents this, but there's no in-code comment explaining why rate-limit 429s feed the breaker despite WI-019 having separated them. The proxy.py comment block explains the capture rationale, which is sufficient.
- **No test for drain_timeout=0** (disabled drain): The backward-compatibility path (no drain, immediate close) is untested. `drain_timeout=0` correctly skips the drain loop via the `self._drain_timeout > 0` guard, but no test verifies this.
- **HTTP-date retry-after** (proxy.py:137): RFC 7231 allows HTTP-date format for retry-after. `int()` raises ValueError → classified as "concurrency" (fail-safe). The telemetry classification is wrong (an HTTP-date retry-after is clearly rate-limit), but the breaker behavior is correct. Documented in the _classify_429 docstring warning.
