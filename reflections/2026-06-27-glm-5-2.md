## On the project

sluice is the first in-path tool in the family — it forwards live traffic, not copies.
That distinction shapes everything: the "inert in-path" guarantee (never read/rewrite
bodies), the disconnect→cancel phantom prevention (the load-bearing detail), and the
single-instance constraint (two sluices each hold their own semaphore, breaking the
account-wide invariant).

The pure-core / shell split is clean and enforced. `control.py` is genuinely pure —
no clock, no I/O, no imports outside stdlib. The breaker state machine I added
(`breaker_on_tick/on_429/on_success`) follows this pattern: time and events are
arguments, transitions return new immutable snapshots. The import-boundary test
catches violations. This is the right architecture.

The reconciliation loop's `record_success` clearing `recent_429s` on HALF_OPEN→CLOSED
transition was a subtle fix — without it, `breaker_on_tick` would immediately re-trip
on the stale 429s. Worth a second look.

## On the work done

**What went well:**
- The controller and usage client were straightforward — pure functions, clear tests.
- The k8s investigation was thorough. Matching the cert-watch deployment pattern
  (security context, Recreate strategy, NetworkPolicy, PDB) was the right call.
- The Dockerfile builds and the container runs correctly — fail-safe behavior verified
  (fake key → 401 → gate tightens to 2 permits, band "low").

**What was awkward:**
- httpx's MockTransport is incompatible with streaming responses. `Response(json=...)`
  sets `is_stream_consumed=True`, breaking `aiter_raw()`. And `content=iter([...])`
  creates a sync-only `IteratorByteStream` that fails the `AsyncByteStream` assertion.
  I had to write a custom `_AsyncMockTransport` that re-wraps content as async generators.
  This is an httpx testing limitation, not a sluice bug, but it cost significant time.
- The ASGI spec uses `"body"` not `"bytes"` for `http.response.body` events. I used
  `"bytes"` initially and all proxy tests failed with empty responses. Easy fix but
  a silly mistake.
- The disconnect test required `asyncio.sleep` delays in the upstream response to give
  the `pump_receive` task time to process the disconnect event between chunks. Without
  delays, all chunks arrive before the disconnect is processed.

**What I'd want a second pair of eyes on:**
- The `pump_receive` task is the single consumer of the ASGI `receive` callable. It
  routes body chunks to a queue and listens for disconnect concurrently. This works
  but the concurrency model is subtle — a bug here could cause deadlocks or missed
  disconnects. The disconnect test passes but covers only one scenario.
- The `_forward` method's gap between "request body fully sent" and "response arrives"
  means a client disconnect during the upstream wait isn't detected until the response
  comes back. The upstream connection is closed at that point, but there's a window.
  This is an acceptable HTTP/1.1 limitation, not a bug, but it should be documented.

## Gaps to flag

- **`proxy.py:149-185`**: The `async with self._client.stream(...)` context blocks
  during the upstream response wait. A client disconnect during this window is not
  detected until the response arrives. The upstream connection is closed at that point,
  but there's a gap where the phantom could persist. Acceptable for HTTP/1.1; would
  need a cancel scope (anyio) to fix fully.
- **`reconcile.py:84-87`**: `record_success` clears `recent_429s` on HALF_OPEN→CLOSED
  transition. This is correct (a successful probe forgives past failures) but subtle —
  if someone refactors the breaker logic, this clearing could be lost, causing
  immediate re-trip. A test asserts this behavior (`test_breaker_success_closes`).
- **`usage.py:148-153`**: The fail-safe reading sets `concurrent_sessions=hard_cap`
  and `priority_low=True`, but `age_seconds=0.0`. The staleness is encoded in
  `fetched_at_monotonic` being very old (`now - 99999`). The reconciliation loop
  computes the age correctly, but if someone changes the loop to not recompute age,
  the fail-safe reading would appear fresh (age=0) despite being synthetic.
- **`test_proxy.py:35-67`**: The `_AsyncMockTransport` is test infrastructure that
  works around httpx's MockTransport not supporting async streaming. If httpx fixes
  this in a future version, the transport can be removed — but until then, it's
  load-bearing for all proxy tests.
- **`deploy/k8s/deployment.yaml`**: Uses `imagePullPolicy: IfNotPresent` with
  `:latest` tag. Updates require deleting the pod or changing the tag. Consider
  pinning to a version tag (`v0.1.0`) once the first release is cut.
