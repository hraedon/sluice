## On the work done

The fixes were small but meaningful:

- **Auth-before-leader** (admin.py): The previous session had leader check before auth, leaking topology to unauthenticated clients. Swapping the order is a clear security improvement with no downside. Both POST and DELETE handlers are now consistent.
- **Body size limit** (admin.py): `_read_body` now caps at 8 KiB and returns 413. This closes WI-025. The disconnect path was also tightened — it now raises `ConnectionError` instead of returning a partial body that could be valid JSON by accident.
- **DELETE audit log** (admin.py): The DELETE path was logging "target reverted" without the previous value, unlike the POST null-revert path which logged "target N -> reverted". Now both paths log the previous value.
- **Unused `config` parameter** (control.py): `validate_target_override(value, reading, config)` never read `config`. Removed it. Both reviewers flagged this independently — the dead parameter suggested a missing validation step (e.g., reject below `min_floor`), but the plan specifies the signature and there's no operational need for it.
- **Docstring fixes**: `BreakerSnapshot` said "Mutable" but it's `@dataclass(frozen=True)`. `history_store.py` said "17 fields" but there are now 22+.

The stale-reading fail-safe concern raised by the adversarial reviewer (Finding #1) is real in principle but I judged it not worth fixing this session. The deprioritization cap IS applied based on the reading's `boxed_until_epoch`, not on freshness — so if the account is in a penalty window, the cap holds even during staleness. The concern is about the *final clamp* using `config.target` instead of `min(config.target, reading.hard_cap)`, but the stale path deliberately distrusts `hard_cap` too. The stale_penalty is flat (1), not proportional, so an overridden target of 8 gives a stale gate of 7 vs a fresh gate of 3 during deprioritization — but this is the same behavior the boot config would produce if target were 8 at deploy time. The fix (clamping to boot_config.target during staleness) would be cleaner but is a larger change.

## Gaps to flag

- `control.py:243` — `effective_permits` stale path clamps to `config.target` not `min(config.target, reading.hard_cap)`. Intentional (don't trust stale `hard_cap`), but with runtime overrides the gap between stale and fresh gates widens proportionally to the override. Not a regression — boot config at the same target would behave identically.
- `admin.py:315` — Unknown JSON fields in POST body are silently ignored (`{"target": 6, "min_floor": 2}` accepts, drops `min_floor`). Could mask operator mistakes.
- `proxy.py:458, 472` — `except (asyncio.CancelledError, Exception): pass` explicitly catches CancelledError. In Python 3.9+ CancelledError is a BaseException, so `except Exception` alone wouldn't catch it. The explicit listing suppresses cancellation signals during entry-task cancellation. Intentional for the entry task, but worth a comment.
- `singleton.py:166` — K8s service account token is read once and cached. After token rotation (~1 hour), the client uses a stale token. Renewal fails, traffic sheds until local expiry triggers. Not a Plan 011 issue but noticed during hygiene review.
- `_is_concurrency_429()` in `proxy.py:135` is dead code — a backward-compatible wrapper that duplicates `_classify_429()`. Only survives because a test directly tests it. Remove both the function and its test.
