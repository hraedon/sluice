# WI-024 capture: genuine umans 429 + penalty box (2026-07-03)

Deliberate over-provision experiment: `--target` was raised 4→8 (`148fd19`),
then 8→10 (`26a7433`, past the reported `hard_cap` of 8) to provoke genuine
upstream concurrency 429s and let the 429 log hook (`294b2d2`) record what
they look like, versus the CDN/gateway 429s that motivated WI-024.

## Timeline (UTC)

| time | event |
|---|---|
| 07:13:49 | pod up with `target: 8` — no 429s observed under team load |
| 07:20:58 | pod up with `target: 10` (past hard_cap 8) |
| 07:23:04.843 | **genuine upstream 429** (log line below) |
| ~07:23 | account enters penalty box |
| 12:23:04.763 | `boxed_until` — **exactly 18000 s (the `requests_window_seconds`) after the 429** |

## The captured 429

```
2026-07-03 07:23:04,843 WARNING sluice.proxy: upstream 429: retry_after='1' classification=rate_limit server=uvicorn
```

Caveat: this is a single observation (n=1); the pod was replaced by a
gitops image-bump rollout minutes later, so no further samples exist and
history persistence was not enabled in the deployment.

## Findings

1. **Genuine app-origin 429s carry no CDN headers.** `server=uvicorn`, no
   `cf-ray` etc. The WI-024 gateway discriminator (CDN header ⇒ gateway) does
   NOT misclassify real umans enforcement 429s — the concern that all
   responses traverse a decorating CDN is empirically false for this path.

2. **Genuine umans 429s arrive with `retry_after: 1`** — a positive value —
   so `_classify_429` labels them `rate_limit` and does **not** feed the
   breaker. The retry-after ≤ 0 heuristic for "concurrency" never fires on
   the real thing. The breaker stayed `closed` through the event that
   preceded a 5-hour box: the exact scenario the breaker exists to prevent.

3. **`rate_limit`-classified 429s increment no counter.** `total_429s`,
   `recent_429s`, and `gateway_429s` all read 0 after the event; only the
   log line records it. Dashboard/metrics are blind to this class.

4. **The penalty box anchors at the 429 with duration = the request
   window** (18000 s to within 80 ms). One over-cap burst cost the account
   5 hours. Consistent with the box being the enforcement response to the
   429-triggering violation, though whether the trigger was concurrency >
   hard_cap or request-window volume cannot be distinguished from n=1.

## Follow-ups

- Reclassify: for umans, small positive retry-after (≈1 s) on a 429 should
  count as concurrency/enforcement signal, not a benign rate-limit window —
  or at minimum feed the breaker and a counter.
- Count `rate_limit` 429s somewhere visible (status + metrics + dashboard).
- Consider honoring `boxed_until` from the 429 response side immediately
  (the usage poll did catch it — `gate_closed_reason: boxed` — verify how
  fast).
- Target reverted to 4 in the same push as this doc.
